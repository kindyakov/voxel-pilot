# GSD Шпаргалка

## Быстрый выбор: что использовать

| Что делаешь?                       | Какая команда?            | Почему?                                             |
| ---------------------------------- | ------------------------- | --------------------------------------------------- |
| Разобраться в новом/старом проекте | `/gsd-map-codebase`       | Покажет архитектуру, не даст AI лесть не в те файлы |
| Начинаю проект с нуля              | `/gsd-new-project`        | Создаст PROJECT.md, ROADMAP.md — фундамент          |
| Хочу обсудить как делать фичу      | `/gsd-discuss-phase N`    | Рождаются требования, ошибки всплывают до кода      |
| Превращаю обсуждение в план        | `/gsd-plan-phase N`       | Создаёт PLAN.md с атомарными шагами                 |
| Делаю фичу по плану                | `/gsd-execute-phase N`    | Волны + атомарные коммиты = чистый код              |
| Проверить что сделано              | `/gsd-verify-work N`      | Тесты + критерии из плана                           |
| Закрыть этап                       | `/gsd-complete-milestone` | Обновляет STATE.md, фиксирует успех                 |
| Быстрая правка/багфикс             | `/gsd-quick`              | Оверхед от плана больше чем от кода                 |
| Прототип "лишь бы работало"        | `/gsd-fast`               | Быстро, но потом придётся чистить                   |

---

## Команды по категориям

### 🚀 Старт проекта

#### `/gsd-new-project`

**Когда:** Первый запуск GSD в пустом или новом проекте.
**Что делает:** Создаёт `PROJECT.md`, `ROADMAP.md`, `STATE.md`.
**Пример:**

```
/gsd-new-project
Название: Minecraft Bot AI
Цель: Автономный бот с LLM-агентом
```

#### `/gsd-map-codebase`

**Когда:** Проект уже существует, нужно объяснить AI его структуру.
**Что делает:** Сканирует код, создаёт карту архитектуры.
**Пример:**

```
/gsd-map-codebase
Фокус: HSM, AI loop, memory layer
```

---

### 💬 Обсуждение и планирование

#### `/gsd-discuss-phase N`

**Когда:** Нужно понять, как реализовать этап N, прежде чем писать код.
**Что делает:** Обсуждение рождает REQUIREMENTS.md, уточняет ROADMAP.md.
**Пример:**

```
/gsd-discuss-phase 3
Как бот должен обрабатывать ошибки pathfinder?
```

#### `/gsd-plan-phase N`

**Когда:** После discuss, когда требования ясны.
**Что делает:** Создаёт `PLAN.md` с XML-шагами, которые можно отслеживать.
**Пример:**

```
/gsd-plan-phase 3
На основе обсуждения в phase-3-discuss.md
```

---

### ⚙️ Исполнение

#### `/gsd-execute-phase N`

**Когда:** PLAN.md готов, нужно реализовать.
**Что делает:** Волнами выполняет каждый шаг из плана, сам делает коммиты.
**Пример:**

```
/gsd-execute-phase 3
```

#### `/gsd-quick`

**Когда:** Мелкая задача: багфикс, одна функция, небольшая правка.
**Что делает:** Выполняет задачу без создания плана.
**Пример:**

```
/gsd-quick
Добавь логирование в функцию connectToServer
```

#### `/gsd-fast`

**Когда:** Прототип, speedrun, proof of concept.
**Предупреждение:** Может создать техдолг.
**Пример:**

```
/gsd-fast
Набросай базовый endpoint для health check
```

---

### ✅ Проверка и завершение

#### `/gsd-verify-work N`

**Когда:** Фаза N выполнена, нужно убедиться, что всё работает.
**Что делает:** Запускает тесты, сверяется с критериями из PLAN.md.
**Пример:**

```
/gsd-verify-work 3
```

#### `/gsd-complete-milestone`

**Когда:** Этап полностью готов и проверен.
**Что делает:** Обновляет STATE.md, фиксирует требования.
**Пример:**

```
/gsd-complete-milestone 3
```

#### `/gsd-ship`

**Когда:** Проект готов к деплою.
**Что делает:** Финальная проверка, обновление всех артефактов.

---

### 🎨 Специализированные команды

#### `/gsd-ui-phase` + `/gsd-ui-review`

**Когда:** Сложный UI с множеством состояний и компонентов.
**Пример:**

```
/gsd-ui-phase
Спроектировать панель управления ботом
```

#### `/gsd-secure-phase`

**Когда:** Критичный код: авторизация, платежи, работа с секретами.
**Пример:**

```
/gsd-secure-phase
Реализовать OAuth для Minecraft сервера
```

#### `/gsd-audit-milestone`

**Когда:** Перед релизом важного этапа.
**Что делает:** Полный аудит кода на безопасность и качество.

---

### 🔧 Управление потоком

#### `/gsd-next`

**Когда:** Хочешь, чтобы AI сам предложил следующий шаг.
**Что делает:** Читает STATE.md и предлагает приоритетную задачу.

#### `/gsd-backlog`

**Когда:** Нужно посмотреть все незавершённые задачи.
**Что делает:** Показывает backlog из STATE.md.

#### `/gsd-threads`

**Когда:** Работаешь с несколькими параллельными задачами.
**Что делает:** Управляет потоками (threads).

## Решения: что выбрать

### Фибоначчи принятия решений

```
ЗАДАЧА ЧТО ДЕЛАТЬ?
│
├─ Меньше 15 минут на код?
│   └─ → /gsd-quick
│
├─ Прототип / PoC?
│   └─ → /gsd-fast
│
├─ Больше 2 часов работы?
│   │
│   ├─ Это архитектурная фича?
│   │   └─ → /gsd-discuss → /gsd-plan → /gsd-execute → /gsd-verify
│   │
│   ├─ Сложный UI?
│   │   └─ → /gsd-ui-phase → /gsd-execute → /gsd-ui-review
│   │
│   └─ Критичный код (безопасность)?
│       └─ → /gsd-secure-phase → /gsd-execute → /gsd-audit
│
└─ Не понимаю архитектуру проекта?
    └─ → /gsd-map-codebase
```

## Артефакты: кто их трогает

| Файл              | Создаёт                            | Обновляет                                 |
| ----------------- | ---------------------------------- | ----------------------------------------- |
| `PROJECT.md`      | `/gsd-new-project`                 | Редко, только при смене курса             |
| `ROADMAP.md`      | `/gsd-new-project`                 | `/gsd-discuss-phase`                      |
| `STATE.md`        | `/gsd-new-project`                 | `/gsd-execute`, `/gsd-complete-milestone` |
| `REQUIREMENTS.md` | `/gsd-discuss-phase`               | `/gsd-discuss-phase`                      |
| `PLAN.md`         | `/gsd-plan-phase`                  | Исполнитель в `/gsd-execute`              |
| `VERIFICATION.md` | `/gsd-verify-work`                 | `/gsd-verify-work`                        |
| `RESEARCH.md`     | `/gsd-plan-phase` (research-агент) | При новом ресерче                         |

## Типичные сценарии

### Сценарий 1: Багфикс

```
1. /gsd-quick
   "Бот крашится при спавне в Nether"

   AI сам находит проблему, фиксит, делает коммит.
```

### Сценарий 2: Новая фича (2+ часа)

```
1. /gsd-discuss-phase 4
   "Как реализовать автоматическую торговлю с жителями?"

2. AI создаёт REQUIREMENTS.md с вариантами

3. /gsd-plan-phase 4
   "На основе обсуждения phase-4"

4. AI создаёт PLAN.md с шагами:
   <step id="1" task="Найти ближайшего жителя" />
   <step id="2" task="Открыть GUI торговли" />
   <step id="3" task="..." />

5. /gsd-execute-phase 4
   AI делает каждый шаг → коммит → следующий

6. /gsd-verify-work 4
   AI запускает тесты, проверяет критерии

7. /gsd-complete-milestone 4
```

### Сценарий 3: Исследование решения

```
1. /gsd-plan-phase 3
   "Как бот должен обходить препятствия?"
   Агент-исследователь создаёт RESEARCH.md,
   анализирует подходы, предлагает решение.

2. Ты читаешь RESEARCH.md → выбираешь подход

3. /gsd-execute-phase 3
```

## Чего НЕ делать

| Ошибка                              | Почему плохо                      | Что делать                            |
| ----------------------------------- | --------------------------------- | ------------------------------------- |
| `/gsd-execute` без плана            | AI делает "как понял" → переделки | Сначала `/gsd-discuss` + `/gsd-plan`  |
| `/gsd-fast` для продакшена          | Техдолг, грязный код              | Используй только для прототипов       |
| Править код в чате во время execute | Ломает атомарность                | Добавь задачу в план или `/gsd-quick` |
| Пропускать `/gsd-verify`            | Код работает "на глаз"            | Всегда проверяй                       |
| Забывать `/gsd-complete-milestone`  | STATE.md устаревает               | Закрывай этапы сразу                  |

---

## Минимальный набор на память

```
1. /gsd-map-codebase      — разобраться в проекте
2. /gsd-new-project        — создать фундамент
3. /gsd-discuss-phase N    — обсудить как делать
4. /gsd-plan-phase N       — написать план
5. /gsd-execute-phase N    — сделать по плану
6. /gsd-quick              — быстрая мелкая задача
```

**80% задач закрываются этими шестью командами.**
