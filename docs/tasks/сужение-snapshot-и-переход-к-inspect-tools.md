# Сужение Snapshot И Переход К Inspect Tools

Этот файл фиксирует текущую задачу по переработке контекста агента, чтобы к ней можно было вернуться позже без опоры на историю чата.

## Текущая задача

Нужно уменьшить объём постоянного контекста, который подаётся агенту в `runAgentTurn`, и привести архитектуру к tool-driven runtime.

Сейчас агент получает слишком богатый `snapshot`, где вместе с runtime-состоянием уже лежат данные о мире: инвентарь, экипировка, видимые блоки, сущности и ближайшие interactable-объекты.

Целевое направление:

- `snapshot` должен содержать только базовый runtime-context.
- Данные о мире должны приходить через inspect-tools по запросу.
- Постоянный prompt не должен подменять инструменты как источник истины.

## Проблема

Текущий `snapshot` размывает границу между состоянием цикла и состоянием мира.

Из-за этого возникают архитектурные проблемы:

- модель начинает отвечать по данным из prompt, а не через tools
- tool-only контракт ослабевает, потому что нужные факты уже лежат в контексте
- сложнее контролировать, какие ответы grounded реальными данными, а какие нет
- prompt разрастается, хотя большая часть данных нужна не всегда

Это противоречит целевой модели работы агента: агент должен узнавать факты о мире через специализированные инструменты, а не через заранее набитый `snapshot`.

## Вариант решения

Рекомендуемый вариант: оставить в `snapshot` только минимальный execution/runtime context.

Оставить в `snapshot`:

- `current_goal`
- `sub_goal`
- `last_action`
- `last_result`
- `last_reason`
- короткий `error_history`
- `health`
- `food`
- `oxygen`
- `position`
- `dimension`
- `active_window_session`:
  - открыто ли окно
  - `window_kind`
  - `block_name`
  - `position`

Убрать из `snapshot`:

- список предметов инвентаря
- экипировку
- видимые блоки
- видимые сущности
- interactables и resources поблизости

Вынести эти данные в tools:

- `inspect_inventory`
- `inspect_window`
- `inspect_entities`
- `inspect_blocks`

Допускается позже объединить `inspect_entities` и `inspect_blocks` в единый `inspect_environment`, если формат останется компактным и понятным.

## Новая архитектурная граница

- `snapshot` отвечает на вопрос: в каком состоянии агент и текущий цикл
- `inspect tools` отвечают на вопрос: что сейчас находится в мире
- `memory` отвечает на вопрос: что мы хотим помнить между задачами

В `memory` не должны попадать текущий инвентарь, ближайшие сущности, текущее окно и другая мгновенная world-state информация.
Туда должны сохраняться только устойчивые и полезные долгосрочные факты.

## Что важно при продолжении

- не пытаться лечить проблему только промптом
- не оставлять богатый `snapshot` и одновременно требовать строгого tool-driven поведения
- сначала зафиксировать новую границу ответственности между `snapshot`, `inspect tools` и `memory`
- только после этого менять `buildSnapshot()` и agent prompt/contracts
