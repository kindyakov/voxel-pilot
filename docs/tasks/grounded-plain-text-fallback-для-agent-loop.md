# Grounded Plain-Text Fallback Для Agent Loop

Этот файл фиксирует текущее решение по обработке plain-text ответов модели без tool call, чтобы к нему можно было вернуться позже без опоры на историю чата.

## Текущая задача

Нужно поддержать модели, которые иногда нарушают tool-only контракт и отвечают обычным текстом вместо `finish_goal`, но при этом не разрушить архитектурную границу между реальными tool-result и выдуманным ответом модели.

## Проблема

Изначально поведение было слишком жёстким:

- если модель не возвращала tool call, ход падал с ошибкой
- даже если до этого уже был корректный `inspect_*` и модель просто забыла вызвать `finish_goal`

Потом был добавлен слишком широкий fallback:

- любой непустой plain text без tool call автоматически считался успешным `finish`

Этот вариант оказался неправильным, потому что агент начинал принимать как истину любой текст модели, даже если он не был grounded инструментами.

Опасный сценарий:

- пользователь спрашивает про инвентарь
- модель не вызывает `inspect_inventory`
- сразу пишет текстовый ответ
- бот публикует это в чат как валидный результат

Такой ответ может быть ложным или галлюцинированным.

## Принятое решение

Fallback должен быть только grounded.

Правило:

- если модель вернула plain text без tool call на первом раунде, такой ответ не принимается
- если модель сначала получила реальные данные через inline tool, а затем вернула plain text без `finish_goal`, такой ответ допускается и публикуется как terminal `finish`

То есть:

- `plain text без tool calls с нуля` -> ошибка
- `inspect_* -> plain text` -> допустимый `finish`

## Архитектурный смысл

Это сохраняет правильную границу:

- агент не может отвечать по неподтверждённой фантазии модели
- агент не падает на моделях, которые после `inspect_*` забывают вызвать `finish_goal`
- tool-runtime остаётся источником истины

Fallback здесь не является заменой tools.
Он является только защитным механизмом для последнего шага, когда данные уже были получены через инструмент.

## Что уже проверено

Покрыты ключевые сценарии:

- plain text с первого раунда -> `failed`
- пустой ответ без tool call -> `failed`
- `inspect_inventory -> plain text` -> `finish`

## Что важно помнить дальше

- не расширять fallback обратно до blanket-режима
- не принимать plain text как успех без grounded tool-output
- если появятся новые inspect-tools, grounded fallback должен работать по тому же правилу: сначала tool data, потом можно принять terminal plain text
