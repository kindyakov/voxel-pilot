# Minecraft Bot: Advanced HSM Architecture

Проект представляет собой высокоавтоматизированного бота для Minecraft, построенного на базе **Hierarchical State Machine (HSM)**. Основная философия архитектуры — разделение ответственности между долгосрочным планированием, выполнением задач и реактивным поведением (выживанием).

## 🧠 Архитектурные концепции

### 1. Иерархическая машина состояний (HSM)
Используется **XState** для реализации параллельной и вложенной логики. Это позволяет боту находиться в нескольких состояниях одновременно.
- **MAIN_ACTIVITY**: Основной контейнер, включающий:
    - `URGENT_NEEDS` (Приоритет 8): Экстренное лечение и питание. Прерывает любые задачи.
    - `COMBAT` (Приоритет 7.5): Логика боя (ближний/дальний бой, защита щитом, отступление).
    - `TASKS` (Приоритет 1-6): Выполнение полезной работы (Mining, Crafting, Smelting).
- **History States**: Бот использует `shallow history`, чтобы возвращаться к выполнению прерванной задачи (например, после того как отбился от зомби).

### 2. Система фильтрации и видимости целей
Бот использует 3-уровневую систему проверки врагов для минимизации вычислительных затрат:
1.  **Distance Check**: Быстрая фильтрация по радиусу.
2.  **Raycast Visibility**: Проверка прямой видимости (глаза бота -> центр цели) для исключения врагов за стенами.
3.  **Pathfind Reachability**: Проверка реального пути через A* (Mineflayer-Pathfinder) с отключенным копанием (`canDig: false`). Результаты кешируются для оптимизации.

### 3. Примитивы и Задачи
- **Primitives**: Атомарные, отменяемые операции (`primitiveNavigating`, `primitiveBreaking`, `primitiveSearchBlock`). Реализованы как `statefulService` с поддержкой `AbortSignal`.
- **Tasks**: Высокоуровневые сценарии, описанные в `TASK_REGISTRY`. Каждая задача определяет:
    - Необходимые параметры (напр. `blockName`, `count`).
    - Используемые примитивы.
    - Предусловия (наличие инструмента, место в инвентаре).

### 4. Интеллектуальная память (BotMemory)
Память бота — это не просто конфиг, а база знаний:
- **World Knowledge**: Координаты и типы контейнеров (сундуки, печи), залежи ресурсов.
- **Experience Analysis**: Статистика успеха каждой задачи (success rate) и среднего времени выполнения.
- **Social Memory**: История встреч с игроками, уровень их дружелюбности и количество взаимодействий.
- **Death Records**: История смертей с записью причин и координат для анализа ошибок.

## 🛠 Технологический стек
- **Mineflayer**: Движок взаимодействия с протоколом.
- **XState v5**: Оркестрация состояний.
- **Winston**: Структурированное логирование в файлы `logs/bot.log` и `logs/error.log`.
- **TypeScript**: Строгая типизация контекста машины и событий.

## 🚀 Команды управления (Разработка)
- `npm run dev`: Запуск через `tsx` с автоматической перезагрузкой при изменениях в коде.
- `npm run build`: Сборка проекта с использованием `tsc` и `tsc-alias` для разрешения путей `@/...`.
- `npm run type-check`: Проверка типов без генерации кода.

## 📁 Структура проекта
- `src/core/`: Ядро бота, инициализация HSM и памяти.
- `src/hsm/machine.ts`: Определение всех состояний и переходов.
- `src/hsm/actors/`: Реализация логики примитивов и сложных акторов.
- `src/hsm/utils/`: Утилиты для HSM (Anti-Loop Guard, Enemy Visibility).
- `src/modules/plugins/`: Адаптеры для плагинов Mineflayer (Pathfinder, PVP, AutoEat).
- `src/utils/`: Общие хелперы (генерация ID, математика Minecraft).

## 🤖 Команда инженеров (Подагенты)
В проекте настроены специализированные подагенты Gemini CLI для глубокой аналитики и автоматизации:
- **`@hsm_architect`**: Проектирование состояний, предотвращение циклов и управление приоритетами.
- **`@minecraft_expert`**: Реализация игровой логики, инвентаря и боевых расчетов.
- **`@bot_debugger`**: Анализ логов (`logs/bot.log`) для поиска причин смертей или сбоев.
- **`@hsm_validator`**: Аудит кода на соответствие стандартам (обязательный `AbortSignal`, регистрация задач).

*Используйте префикс `@имя_агента` в запросе, чтобы привлечь специалиста к задаче.*

## ⚠️ Правила разработки
1.  **Safety First**: Любое действие должно быть прерываемым через `abortSignal`.
2.  **Context isolation**: Не изменять `bot.entity` напрямую, использовать методы примитивов или плагинов.
3.  **Anti-Loop**: HSM защищена `AntiLoopGuard`, который остановит бота при слишком частых переходах между состояниями (защита от бесконечных циклов).
